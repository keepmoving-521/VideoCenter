# R22-R23：防止重复解析与解析器单元测试

## R22 进行中任务合并

结果缓存只能避免“已经完成”的重复解析。R22 进一步处理同时到达的相同请求：

```text
请求 A ─┐
请求 B ─┼── 同一个解析任务 ── 共享结果
请求 C ─┘
```

判定相同请求的键与解析缓存一致：

- 解析器名称；
- 规范化后的资源 URL；
- 首选语言。

第一个请求创建实际解析任务，后续并发请求等待该任务，不再重复访问目标网站。等待方日志使用 `parse_event=duplicate_wait`，并通过 `shared_parse_task_id` 关联实际执行的任务。

任务成功后结果进入缓存；任务失败后进行中状态立即清理，下一次请求可以重新解析，不会被永久锁住。等待方取消请求不会取消共享解析任务。

当前任务合并范围为单个应用进程，多进程部署时需要使用 Redis 分布式锁或任务队列实现跨进程合并。

## R23 单元测试

新增独立的解析器单元测试目录 `tests/unit/`，覆盖：

- 并发相同请求只调用一次解析器；
- 共享任务失败后能够重新解析；
- 等待任务与实际任务的日志关联；
- 重复 Open Graph 视频标签；
- JSON-LD 非法数据容错；
- 视频和字幕地址去重；
- 上映年份边界；
- JSON-LD、Open Graph、HTML 标题优先级。

测试全部使用内存 HTML 和可控的假解析器，不访问真实网站，因此执行稳定且不受外部网络影响。

本次迭代不修改数据库结构，不需要新增 Alembic 迁移。
