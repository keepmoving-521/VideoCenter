# Q13：应用重启后恢复任务

应用启动时会通过统一恢复协调器处理未完成后台任务，不再由启动入口分别调用四套恢复逻辑。

## 恢复顺序

启动恢复分为两个阶段：

1. 审计重启前仍为 `running` 的统一任务；
2. 恢复下载、扫描、分析和 HLS 业务任务并重新排队。

先审计再启动线程，可以避免把刚刚恢复执行的任务误判为异常中断。

## 状态审计

对于有业务来源的统一任务，系统会读取真实业务任务状态并修复统一任务：

- 下载任务与 `download_tasks` 对齐；
- 扫描任务与 `scan_tasks` 对齐；
- 分析任务与 `analysis_tasks` 对齐；
- HLS 任务与 `hls_tasks` 对齐。

审计会写入 `recovery_audit` 日志。

以下运行中任务无法自动恢复，会标记为 `failed`：

- 进程内执行、没有持久化业务来源的资源解析任务；
- 通用任务；
- 关联业务任务已经丢失的统一任务。

错误码为 `TASK_INTERRUPTED_BY_RESTART`，同时写入 `recovery_failed` 日志，避免任务永久停留
在运行中。

## 业务任务恢复

- 下载中的任务重置为等待，并按优先级重新进入下载队列；
- 等待或运行中的扫描任务重置进度并重新启动；
- 等待或运行中的分析任务重置进度并重新启动；
- 等待或运行中的 HLS 任务重置进度并重新进入转码队列；
- 暂停的下载任务保持暂停，不会自动恢复下载。

启动日志会汇总各类型恢复数量、状态修复数量和无法恢复的中断任务数量。

## 数据库

本次复用统一任务和任务日志表，没有修改数据库结构，不需要新增 Alembic 迁移。
